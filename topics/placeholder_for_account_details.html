UI control Business logic

Decoupling the UI presentation logic from the business logic means:

  • to implement any UI independent logic outside an UI layer

  • to implement any logic which can be useful outside UI, somewhere outside an UI layer

  • to decouple the system from specific UI technology

For example, designers often want or need to have: the UI controlling logic on the server; the UI uses HTML or JavaScript on the client side; the UI controlling logic to become subject of an automated test.

The idea is that the UI presentation logic should not touch the business rules at all. The business logic sits at the back-end and the UI presentation logic sits on the outside. UI presentation logic should just transfer commands and display any updated output.

The problem is that it's quite difficult to imagine in practice. The UI presentation will likely implement conditional rendering, and much of that conditional rendering is a business rule. We MUST unify approaches to decoupling the UI presentation logic from UI business logic.

Decoupling system from specific UI technology The decoupling system from specific UI technology can be realized by introducing an extra view model layer or presenter layer, which contains the UI controlling logic, but communicates with the UI through an interface (and so keeps the UI technology exchangeable).

What Tasks and/or Steps a user can access depending on his role. Can access. in this case, means restrict access to not allowed data and functions or based on the data that the user's role assigned tasks work with, the system must restrict access to features. Restricting access is an act. Since processing the user's role is business logic of restricting access to features, it must not be implemented in the UI. It needs to be implemented in the core of the application (back end). The functions/methods themselves should throw an error if a user is not allowed to use them.

UI control Presentation logic

Showing a list differently is not business logic, it is presentation logic. It should not be implemented at the back end (where you process the data to display).

While deciding how to display a list has zero business logic, there are some forms of conditional rendering that touches business logic.

Showing or hiding what the user is allowed to do is a business logic in the back end that returns what the UI controls are supposed to render as navigation items. However, it may be implemented on UI layer so that UI presentation logic can show/hide navigation UI controls for tasks and/or their steps allowable to the user. We MUST unify approaches to decoupling the UI presentation logic from UI business logic.

UI presentation logic can show/hide navigation UI controls for tasks and/or their steps allowable to the user. In this case, showing or hiding what the user is allowed to do is presentation logic. It can be implemented purely in UI based on the user's role. In this case you’re implementing business logic as presentation logic and, it's OK as far as it's not the only way you're restricting user's access to features.

A more modular approach, however, is to have business logic in the back end return what the UI is supposed to render as navigation items.
